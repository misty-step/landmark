LANDMARK RUST CLI / RELEASE KIT CONTRACT

Make landmark run --dry-run emit a schema-valid release-kit plan.

The runtime should turn release facts into a typed final-mile artifact graph: Landmark-owned outputs, adapter-owned outputs, provenance, approvals, blockers, and producer handoff contracts. Keep the core as release intelligence; do not add media production, CMS publishing, or bespoke producer execution.

CHOSEN DESIGN

Extend the Rust runtime's release evidence path to derive and write a release-kit JSON artifact from the existing release context and importance classification.

QUALITY SYSTEM

Tier: high-stakes for release automation public surface.

Proof: CLI dry run transcript, schema validation through existing contract checks, replay scenario coverage, and fresh artifact-only critic.

Premise source: Powder card landmark-021

STANDARDS

Rust owns runtime behavior. JSON contracts stay schema-backed. GitHub behavior stays adapter-specific. No Node, shell, or Python orchestration enters the release-kit core.

ACCEPTANCE

run --dry-run produces or prints a landmark.release-kit.v1 artifact with artifacts, producer contracts, provenance, approvals, and status.

VERIFY

cargo test --locked, cargo run --locked -- check-action-contract, bin/replay-action, focused dry run, then bin/gate.

COMMUNICATE

Report plan opened, failing tests, implementation surface, replay evidence path, critic result, gate, final git state, and residual risk.

STOP IF

Schema needs a breaking change, replay fixtures contradict the backlog oracle, or implementation starts executing producer work instead of planning it.

CURRENT STATE

CHANGE SHAPE

Add typed Rust structs and planner logic near the release context/evidence path, wire output paths and dry-run stdout, then extend replay tests.

PROOF SURFACE

Evidence lives under .landmark/run and .landmark/replay; schema truth lives in schemas/release-kit.v1.schema.json.

SCOPE

Goal: runtime generation of the release-kit graph.

Invariants: dry-run remains non-mutating; GitHub mutation requires explicit adapter flags; fallback release publishing semantics stay intact.
